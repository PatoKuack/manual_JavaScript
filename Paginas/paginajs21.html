NPM.

NPM o node package manager, es un gestor de paquetes, el más popular que tiene JavaScript, donde se encuentra una gran cantidad de recursos para poder implementar en los proyectos. También permite crear paquetes propios y compartirlos con toda la comunidad.

Instalación de Node en Windows.

Para instalar NPM en Windows es necesario ir a la página de node: nodejs.org, para instalar node y la versión de NPM.

En la página se encuentran dos opciones:

  • La LTS que brinda un soporte a todos los features (características) que están añadidos en esa versión y es mejor para entornos de desarrollo y producción.
  • La Current (que es la que se descargará e instalará) contiene todos lo features que han salido hasta ahora.

Una vez instalado (dando en toda la instalación NEXT), se procede a abrir el entorno del software que es una terminal de comandos. En la terminal se utilizarán las siguientes instrucciones:

node -v
Muestra la versión que se tiene de node.
npm -v
Muestra la versión que se tiene de npm.
npm install -g npm@latest
Comprueba si existe una versión más resiente de npm y si es así se instala en el sistema.

Inicializar proyecto.

Se crea la carpeta que contendrá al proyecto usando el comando mkdir nombre, después se accede a esa carpeta usando el comando cd nombre y ya adentro se ejecutan los siguientes comandos:

git init
Inicializa el repositorio local del proyecto en github para tener el control de cambios y versiones en nuestro proyecto.
npm init
Crea el archivo package.json con el cual se tendrá una configuración establecida, una descripción del proyecto y ciertos valores que se necesita.

Al ejecutar este comando se mostrará en la terminal una introducción sobre el archivo y pedirá los siguientes datos:

package name
Pide el nombre del proyecto, generalmente es el nombre de la carpeta. Si se da enter sin especificar un nombre se colocará el nombre por default mostrado entre paréntesis.

*ATENCÓN, el nombre debe ser original, si existe un paquete con el mismo nombre de otro paquete subido por otro usuario en el manejador de paquetes, marcará un error 403.

versión
Pide la versión con la que se inicia el proyecto, generalmente aparece 1.0.0 entre paréntesis que si se da enter se guarda esa versión inicial, aunque se puede colocar manualmente la versión deseada.
description
Pide una descripción breve del proyecto.
entry point
Pide el punto de acceso al proyecto, se puede dar enter para dejar el archivo que se sugiere entre paréntesis o se coloca la ruta o nombre del archivo que se desee (ejemplo: src/index.js).
test command
Pide comando para gestionar los test o scripts que serán importantes al correr comandos específicos a la hora de trabajar en el proyecto. Se puede dar enter sin especificar alguno y configurar esta parte más adelante.
git repository
Pide el repositorio de github u otro servicio. Se puede dar enter sin especificar alguno y especificarlo más adelante
keywords
Pide las palabras claves del proyecto, las palabras que más se cree que se utilizarán en el proyecto.
author
Pide el nombre del autor del proyecto y el correo electrónico entre signos de menor y mayor que <correo@mail.com>
license
Pide la licencia que tendrá el proyecto, como la del MIT que permite trabajar con proyectos opensource.

Una vez que se termina de ingresar los datos, se muestra una previsualización del archivo package.json. Si todo está correcto con el archivo se da enter o se escribe la palabra yes

El archivo package.json es el documento principal donde se especifica toda la configuración y los recursos informativos del proyecto.

npm set init.author.name "Fulanito"
Añade en la configuración de npm el nombre del autor.
npm set init.author.email "correo@mail.com"
Añade en la configuración de npm el e-mail del autor.
npm set init.license "MIT"
Añade en la configuración de npm la licencia deseada.
npm init -y|-yes
Crea un package.json con los datos por default y los añadidos al npm.

Instalación de dependencias.

Lo primero que se hace es crear la carpeta que contendrá el proyecto generalmente llamado src.

Se crea el archivo index.js con el comando touch index.js que es el archivo que consumirá los recursos que se han descargado.

Desde la raíz del proyecto se instalan las dependencias o recursos que se utilizarán en el proyecto.

Una dependencia de software es: un paquete, biblioteca, o conjunto de estos, necesarios para que otro paquete pueda instalarse y ejecutarse.

Al momento de agregar dependencias se crea automáticamente una carpeta en el proyecto llamada "node_modules", en esta carpeta es donde se instalan los módulos que se agregan al proyecto. Esta carpeta es importante para que el proyecto funcione, pero no debe de ser enviada a ningún repositorio por lo que se debe ignorar apenas se crea. Para ignorarla se agrega en el archivo .gitignore el texto node_modules/

npm install paquete --save-dev
Con esta instrucción se instala un paquete y con la palabra anexa "--save" se indica que el paquete se agregará al proyecto que se subirá al servidor, si no se agrega la palabra "--save" no se subirá al servidor. La palabra anexa "-dev" indica que el paquete es necesario en el entorno de desarrollo, si no se agrega la palabra "-dev" se indica que se utilizará el paquete sólo en el entorno local.
npm i paquete -S-D
Es una forma abreviada de pedir que se instale una dependencia a npm y que se guarde como parte del proyecto y como una dependencia de desarrollo.
sudo npm istall -g nodemon
Permite generar un demonio que se encarga de escuchar un cambio de algún valor sin detener algún proceso que se esté ejecutando. Este paquete funciona de manera global, en todos los archivos.

Demonio: Servicio que corre en segundo plano en el sistema

npm list -g --depth 0
Muestra un listado de paquetes instalados de forma global (depth es la profundidad que va a buscar dentro de los paquetes).
npm install paquete -O
Permite instalar de forma opcional un paquete.
npm install paquete --dry-run
Simula la instalación del paquete especificado. Este flag indica que el paquete no va a ser instalado dentro del proyecto, simplemente es una simulación y nos muestra el output como si fuese instalado, después de esto nosotros decidimos si la instalamos o no.
npm install paquete -f | --force
Instala el paquete de forma forzada obligándolo a que sea desde el último recurso o versión y desde el servidor de NPM.
npm install
Vuelve a instalar todas las dependencias que están en el package.json
npm install paquete@versión
Instala el paquete con la versión especificada, esto puede ser necesario cuando tenemos que darle mantenimiento a una app con una versión anterior de algún paquete o quizás la versión actual es beta (ejemplo: npm install json-server@0.15.0).

Actualizar paquetes.

npm list
Muestra un listado de todo lo instalado, los paquetes con su jerarquía.
npm outdate
Muestra los paquetes que están desactualizados.
npm outdate --dd
"--dd" devuelve u output o salida más detallada del comando, en este caso muestra los paquetes desactualizados con información más detallada con la versión que se tiene de node y de npm, la sesión que se está utilizando y donde se encuentran ciertas configuraciones del proyecto.
npm update
Actualiza los paquetes que no están en la última versión.
npm install paquete@latest
Actualiza un paquete específico.

Eliminar paquetes.

npm uninstall paquete
Elimina un paquete de node_modules y del archivo package.json
npm uninstall paquete --no-save
Desinstala un paquete de node_modules, pero no del archivo package.json

Versionado semántico.

En el archivo package.json en la parte de dependencias se tienen tres valores numéricos separados por un punto, éstos valores se encuentran al lado de cada paquete y se establecen dentro del versionado. Cada uno de los dígitos tiene un significado:

  • Primer dígito: son cambios mayores (versión 1, versión 3, etc.).
  • Segundo dígito: añaden funcionalidades que no representan un gran paso en el paquete como para considerarlo como una nueva versión.
  • Tercer dígito: estos son patch (parches), bug fixes o cambios menores.

Símbolos.

^
Cuando se mantiene el símbolo ^ (acento circunflejo o caret) dentro de la configuración del package.json, se garantiza que cuando se haga una actualización o un cambio, se actualizan sólo los cambios menores y los parches o bug fix de este paquete. Si se desea mantener el control sobre las actualizaciones garantizando que se queden en cierta versión lo mejor es elimina el caret (^).
~
Cuando se coloca el símbolo ~ (virgulilla o tilde) se recibirán actualizaciones o cambios que son parch o bug fixes.
<
Se reciben versiones menores a la indicada.
<=
Se reciben versiones menores o iguales a la indicada.
>
Se reciben versiones mayores a la indicada.
>=
Se reciben versiones mayores o iguales a la indicada.
 "dependencies": {
  "json-server": "0.15.1",
  "moment": "^2.26.0",
  "date-fns": "~2.14.0",
  "react": ">=16.12.0"
 }

A partir de la versión 5 de npm se crea un archivo llamado package-lock.json que permite tener configuraciones visibles del proyecto como las versiones, los paquetes y las dependencias que se encuentran dentro del proyecto, se puede compartir este documento con los demás desarrolladores y garantizar que las versiones que se están instalando sean las correctas, también sirve para que el servidor instale de manera automática todas estas dependencias cuando los proyectos estén en la nube.

Scripts.

Los scripts en NPM son comandos que pueden ser establecidos para ejecutarlos de forma nativa dentro de nuestra terminal con el fin de obtener salidas específicas. Se puede crear la cantidad de scrips que se desee.

 "scripts": {
  "build": "webpack --mode production",
  "hola": "node"
 }

Para ejecutar estos scripts se utiliza desde la terminal la instrucción: npm run script

También se puede especificar scripts con el prefijo “pre” que se ejecutarán automáticamente antes del comando que que se ejecutó. Por ejemplo, si se define el comando build y prebuild, cuando se corra npm run build, el comando prebuild se ejecutará primero. Esto sirve para poder ejecutar tareas que hagan algún tipo de preparación necesaria para correr el comando principal. Sin embargo, hay que hacer notar que si el comando pre falla (retorna un valor que no es 0) el comando principal no se ejecutará. Si el proceso de preparación no se realiza de forma exitosa, puede que genere problemas al querer ejecutar la tarea principal.

En algunas ocasiones, sin embargo, la tarea previa puede fallar sin que eso afecte la ejecución de la tarea principal. En esos casos se puede usar || exit 0 para retornar 0:

 "presass-build": "(rm css/*.css; rm css/*.css.map) || exit 0",
 "build": "webpack --mode production",

Solución de problemas.

Cuando se trabaja con proyectos que están usando NPM aparece una gran cantidad de posibles errores. Estos errores pueden ser desde la configuración, desde el sistema operativo, por espacios, por no haber configurado correctamente GitHub, no haber establecido bien los datos del package, haber dejado un typo o algún elemento extraño dentro de esta configuración, así como una serie de errores que pueden generarse sin estar ligados directamente a NPM.

Para solucionar estos errores se debe leer detenidamente lo que nos imprime la consola y si la instrucción fue correctamente escrita.

En caso de que los archivos de node_module no estén bien instalados, se tenga un valor corrupto o se tenga una versión anterior, se puede eliminar la carpeta o limpiar el cache existente.

# Para limpiar el cache del sistema:
 npm cache clean --force
# Para verificar que se limpió:
 npm cache verify
# Para eliminar la carpeta node_modules:
 rm -rf node_modules
# Para reinstalar todas la dependencias:
 npm install
# Otra alternativa para eliminar de forma segura una carpeta (sobre todo en sistemas Window) es instalando el siguiente paquete con permisos de administrador:
 sudo npm install -g rimraf
# Después se elimina la carpeta node_modules:
 rimraf node_modules
# Y se instala de nuevo paquetes
 npm install

Seguridad.

Se debe garantizar de que el proyecto no contenga softwares maliciosos y que cumpla con las normas de trabajo. Para evitar ésto se puede utilizar NPM para analizar el proyecto y detectar vulnerabilidades.

npm audit
Sirve para ver las vulnerabilidades del proyecto.
npm audit --json
Genera un archivo .json con información más detallada de lo que está pasando con los paquetes instalados.

Una ves sepamos cual es la vulnerabilidad se procede a actualizar cualquiera de los paquetes.

npm update paquete --depth 2
Instala todas las dependencias del archivo en un nivel de profundidas de 2.
npm audit fix
Soluciona las vulnerabilidades del proyecto y actualiza a la última versión los paquetes con las dependencias que requieren.

Se puede correr de nuevo el comando npm audit para verificar que ya no haya vulnerabilidades. Si aún hay vulnerabilidades, es recomendable actualizar uno por uno, paquete por paquete.

snyk.io es una herramienta que garantiza estar siempre actualizados con las dependencias del proyecto.

Uso de paquetes NPM

Los paquetes que se instalan y sobre todo de los que dependerá nuestro proyecto deben de tener un buen soporte, buenas actualizaciones y haber sido ampliamente testeados por muchos usuarios.

Si se desea verificar el código del paquete se debe abrir el archivo index.js en el repositorio de github.

En esta página se puede buscar un paquete y revisar su versión, última actualización, numero de descargas y más. También se puede verificar cuantas dependencias tiene el paquete, las dependencias son los paquetes que utiliza un paquete para funcionar y al instalar un paquete puede que se tengan que instalar varios más por estas dependencias y por las dependencias de las dependencias.

Para poder utilizar el paquete en un proyecto se debe hacer de la siguiente forma:

 const función = require('nombre-del-paquete');
 función( parámetros );

Creación de paquetes.

Primero se debe crear la carpeta donde se alojará el proyecto y desde la terminal se posiciona en esa carpeta. Aún desde la terminal se inicializa el repositorio en git (git init), se inicializa la configuración del proyecto con NPM (npm init) y se abre el editor de código.

En el editor se crea por buenas prácticas una carpeta en donde se alojará el código creado, generalmente esta se llama src y dentro de esta carpeta se crea el documento index.js que es donde se alojará el código:

 // Se crea el código del proyecto:
 const proyectoPaquete = () => {...}
 // Se exporta:
 module.exports = { proyectoPaquete }

Se crea una carpeta llamada bin en la misma carpeta donde se encuentra la "src". En esta carpeta bin se creará el archivo global.js que es en donde se estará definida la configuración para que el paquete que se está creando en este proyecto sea un paquete que se instala de forma global.

 // Se establece que es un elemento que se ejecutará dentro del sistema bash.
 #!/usr/bin/env node
 // Se crea una variable que pida como requerimiento el archivo index.js
 let proyectoPaq = require('../src/index.js');
 // Se declara la ejecución del paquete:
 proyectoPaq.proyectoPaquete();

Desde el archivo package.json se creará al final del documento un apartado llamado bin en donde se agregará la configuración para llamar al paquete registrando la palabra que se utilizará para llamarlo desde el archivo global.js

Para que sea un paquete global se coloca en "true" el valor "preferGlobal".

  ...
  "author": "... <...>",
  "license": "...",
  "bin": {
   "proPaquete": "./bin/global.js"
  },
  "preferGlobal": true
 }

Se prueba el paquete de manera local para garantizar que cumple con el funcionamiento que se desea, para esto se utiliza un recurso de NPM para hacer un link de este proyecto para implementar el paquete en otros proyectos o instalarlo de manera global. Desde la terminal y estando posicionado en la carpeta del proyecto se ejecuta el comando:

# Si trabajas en una distribución linux, utiliza "sudo".
 sudo npm link

Con link se hace una referencia al paquete del proyecto instalándolo en el sistema como si hubiera sido instalado desde los servidores de NPM. Para comprobar se encuentra en el sistema se ejecuta desde la terminal con la palabra reservada definida en el archivo global.js (en este caso en proPaquete).

Si modificamos nuestro paquete se puede actualizar en el sistema para seguir probando su funcionamiento con la siguiente instrucción:

 # Desde la terminal y posicionados en la carpeta del proyecto:
 sudo npm install -g ruta del proyecto
 # También se puede utilizar una virgulilla para las distribuciones de linux:
 sudo npm install -g ~

Para poder mandar el paquete a NPM y poderlo publicar, primero se tiene que crear una cuenta o iniciar sesión en ella en la página de NPM, después se tiene que loggear en esta cuenta desde la terminal con el comando npm adduser, se pedirán nombre de usuario, contraseña e e-mail.

Para publicar el paquete se utiliza la instrucción npm publish la cual lee la configuración del archivo package.json y establece el manejador de paquetes que sería npmjs.org

En la página web de NPM, en una sección llamada "packages" en la parte de la cuenta, se puede encontrar el paquete

Después de loggear el paquete ya se puede descargar y utilizar por cualquiera que tenga NPM instalado, pero se deben implementar algunos atributos en el paquete para que sea óptimo para otros usuarios como crear un buen README.md, conectarlo a un repositorio y mantener actualizado el paquete.

En el README.md se explica lo que hace el paquete, como se instala, los comandos que contiene y una invitación a que contribuyan a mejorar el proyecto. El README.md debe estar en inglés.

Una de las opciones que muestran los paquetes una vez que están publicados es la del repositorio, los errores que tiene, si alguien los reportó y las correcciones que se le han hecho al proyecto. Es por eso que es importante conectar el proyecto con un repositorio en la nube.

 # Ejemplo de un repositorio en GitHub
 # Desde la terminal y posicionados en la carpeta del proyecto:
 git init
 git remote add origin https://github.com/....git

Se debe colocar una buena descripción en inglés en el archivo package.json

Se debe abordar en el archivo package.json la parte de pruebas o test (pruebas unitarias con Jest).

Una vez construido el proyecto y asignado al repositorio se utiliza el comando npm init para que el proyecto genere en el archivo package.json la sección en donde se encuentra el repositorio, en donde se reportarán los bugs y la página principal del proyecto (también se pueden crear estas secciones en el archivo a mano).

Después se realiza un commit para actualizar los cambios en el proyecto en el repositorio y se actualiza la versión dependiendo de si es un parche, un cambio menor o uno mayor y se publica de nuevo el paquete para que este disponible la nueva versión con todos sus cambios para los usuarios:

 npm version patch | minor | major
 sudo npm install -g npm
 npm publish
