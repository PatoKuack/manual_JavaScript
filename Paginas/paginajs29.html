Webpack

Es un empaquetador de archivos para aplicaciones JavaScript modernas, totalmente configurable y se conoce el origen y todo se compila en un único archivo.

Crea una gráfica de todas las dependencias de la aplicación. Tiene un archivo de configuración, denominado webpack.config.js, donde se define todo el proceso de construcción en un objeto JS.

Webpack permite trabajar con HTML, CSS, JavaScript, archivos estáticos, imágenes y fuentes, también permite gestionar dependencias, ejecutar tareas, convertir archivos y trabajar en módulos obteniendo un código separado en desarrollo para hace pruebas del proyecto y otro en producción y por último permite tener módulos de JS en formato AMD, Common JS y ES15.

Webpack tiene 4 conceptos clave:

  1. Entry: Indica cuál es el punto(s) de entrada.
  2. Output: Indica cuál es el punto(s) de salida. Este archivo es el bundle o el archivo de salida, sería una caja en donde se empaqueta toda la aplicación, normalmente este archivo final se crea en una carpeta llamada dist.
  3. Loaders: Realizan transformaciones en los archivos. Permite hacer un bundle de cualquier recurso estático más allá de JavaScript.
  4. Plugins: Realizan acciones en los archivos. Extienden recursos para añadir configuraciones y particularidades de recursos externos.
Module bundlers
Son herramientas de frontend que permiten usar archivos con módulos JavaScript (entre otras características) y convertirlos a un JavaScript el cual el navegador pueda entender (como webpack).

Instalación de webpack.

El proyecto inicialmente tiene en la raíz 3 archivos: .gitignore, README.md y LICENSE.

Se deben crear las siguientes carpetas:

  • /src En ella se alojará todo el código del proyecto.
    • index.js
    • /components En ella se alojan los conponentes que se crean en react.
      • App.jsx
    • /assets
      • /images
      • /fonts
    • /utils En ella se alojarán los archivos .js.
      • document.js
    • /styles En ella se alojarán los estilos.
      • main.css
  • /public En ella se alojan los archivos de los que dispone el proyecto.
    • index.html

Al trabajar con react se tienen que editar los archivos index.js, index.html y App.jsx.

// Se inicializa un repositorio de forma local con git.
git init
// Se inicializa el proyecto con npm (al ejecutar la instrucción se creará el archivo package.json).
npm init
// Se puede instalar react si es que se utilizará en el proyecto más adelante.
npm install react react-dom -S
// Se instala webpack como una dependencia de desarrollo (no de forma global).
// webpack-cli permite usar comandos dentro de la terminal para traajr con webpack.
npm i webpack webpack-cli -D
// Se ejecuta webpack con npx.
// npx ejecuta paquetes directamente de npm, este viene instalado en npm.
npx webpack

Al ejecutar npx webpack se crea una carpeta llamada dist que contendrá un archivo llamado main.js en donde se aloja el código compilado de la ruta src/index.js, también se crea el archivo package-lock.json y la carpeta node_modules.

Por defecto webpack al compilar el proyecto, lo setea (o compila) en el modo “production” implícitamente, pero es posible definir explícitamente el modo que se desea con las siguientes instrucciones:

// El modo production deja el código comprimido, limpio y optimizado para utilizarse.
npx webpack --mode production
// El modo development deja el código mas legible para los desarrolladores pero con comentarios.
npx webpack --mode development

Al compilar el proyecto se obtiene entre tanto, un archivo index.html en el cual se visualiza el resultado de toda la compilación del proyecto.

webpack.config.js

Este es el archivo de configuración, establece la configuración y elementos que se utilizan. Este archivo se crea en la raíz del proyecto y en él se especifica los siguiente:

// webpack.config.js
// path es donde estará la carpeta donde se guardará los archivos.
const path = require('path');
// Se abre el objeto en donde se colocarán las configuraciones.
module.exports = {
 // Con entry se especificar el punto de entrada de la aplicación.
 entry: "./src/index.js",
 // Con output se especifica hacia dónde se envía lo que prepara webpack.
 output: {
  // Con path.resolve se especifica dónde se hubica el proyecto.
  path: path.resolve(__dirname, "dist"),
  // filename le asigna el nombre al archivo final, puede colocarse el main.js o la estructura con el uso del hash:
  // filename: "main.js"
  filename: "[name].[contenthash].js"
 },
 resolve: {
  // En extensions se colocan las extensiones que se tendrán en el proyecto para que webpack los lea, en el ejemplo solo se utiliza la extensión .js.
  extensions: [".js", ".jsx"]
 },
}

Una vez configurado el archivo de configuración, se compila con la instrucción:

npx webpack --mode production --config webpack.config.js

Se puede colocar en el archivo package.json en la sección de scripts la instrucción:

// package.json
scripts": {
 ...,
  "build": "webpack --mode production"
},

Para facilitar la compilación del archivo de configuración, permitiendo colocar npm run build en vez de npx webpack --mode production --config webpack.config.js

Se asegura q todo este bien con la instrucción:

npx webpack --mode production

Loaders y plugins.

Webpack solamente entiende JavaScript y JSON por lo que los loaders ayudan procesando archivos de otros tipos para convertirlos en módulos válidos para webpack que serán consumidos por la aplicación que se cree y serán agregados como dependencias.

En alto nivel, los loaders poseen 2 configuraciones principales:

  1. test - propiedad que identifica cuáles archivos deberán ser transformados.
  2. use - propiedad que identifica el loader que será usado para transformar a dichos archivos.

Los plugins son utilizados para extender tareas específicas, como la optimización de paquetes, la gestión de activos y la inyección de variables de entorno. Una vez importado el plugin, se pueden personalizar.

Loader Babel.

Babel ayuda a que el código JavaScript sea compatible con todos los navegadores.

babel-loader es un loader que permite usar Babel con webpack, para instalarlo se utiliza la siguiente instrucción desde la terminal:

npm install -D babel-loader @babel/core @babel/preset-env @babel/preset-react @babel/plugin-transform-runtime
@babel/core permite utilizar babel en general.
@babel/preset-env permite utilizar las últimas características de JavaScript.
@babel/preset-react permite utilizar las la sintaxis de react y poder identificar los archivos .jsx.
@babel/plugin-transform-runtime permite trabajar con asincronismo, por ejemplo con async y await.

Como se agrega la instrucción -D, las dependencias se colocan en el área de devDependencies del archivo package.json

Después se crea, en la raíz del proyecto, un archivo de configuración de Babel con el nombre .babelrc (el punto en el nombre indica que será un archivo oculto, esto se utiliza para los archivos de configuración), en el archivo se crea un objeto donde se añaden las configuraciones como presets para traajar con JavaScript moderno y plugins que utilizará Babel.

// .babelrc
{
 "presets": [
   "@babel/preset-env",
   "@babel/preset-react"
 ],
 "plugins": [
   "@babel/plugin-transform-runtime"
 ]
}

Para comenzar a utilizar webpack se debe agregar la siguiente configuración en webpack.config.js:

// webpack.config.js
module.exports = {
 ...,
 resolve: {
  ...
 },
  module: {
  rules: [
   {
    // Con test se indica que el loader utilizará archivos con las extenciones .mjs o .js
    test: /\.(mjs|js|jsx)$/,
    // Con exclude se indica que se excluirá lo que provenga de la dirección node_modules.
    exclude: /node_modules/,
    // Use es un arreglo u objeto donde se especifica que loader se aplicaran.
    use: {
     loader: "babel-loader"
    },
   }
  ]
 }
}

Por ultimo, se compila webpack para añadir las configuraciones de Babel, utilizando la instrucción definida npm run build o la instrucción de default npx webpack --mode production --config webpack.config.js

Plugin HTML.

htmlWebpackPlugin es un plugin para inyectar javascript, css, favicons y facilitar la tarea de enlazar los bundles al template HTML. Para utilizarlo primero se debe instalar el plugin desde la configuración de webpack a traves de la siguiente instrucción en la terminal:

npm i html-webpack-plugin -D

Después se agrega en el archivo de configuración webpack.config.js de la siguiente forma:

// webpack.config.js
const path ...;
const HtmlWebpackPlugin = require('html-webpack-plugin');
module.exports = {
 ...,
 module: {
  ...
 },
  plugins: [
  // Se agrega un nuevo plugin utilizando la constante declarada al inicio del documento y dentro se coloca un objeto para definir las configuraciones de este.
  new HtmlWebpackPlugin({
   // Se especifica que se autorice la inyección de los elementos.
   inject: true,
   // Se indica la ruta del template.
   template: './public/index.html',
   // Se indica el nombre final del archivo que por estandar es index.html.
   filename: './index.html'
  })
 ]
}

Al compilar el proyecto con webpack se creará en el archivo index.html, al final de la etiqueta head una etiqueta script que llama al main.js (si se había creado otro script se podría borrar si es redundante el uso, es decir, si es llamado otra vez el main.js o el index.js).

Loader CSS.

Se instala el loader para css y si se desea se instala el loader para sass y para stylus.

npm install mini-css-extract-plugin css-loader style-loader sass sass-loader -D
npm i -D stylus stylus-loader

Después se elimina del archivo index.html la etiqueta link que vincula el main.css con el proyecto.

En el archivo index.js se importan los estilos.

// index.js
import './styles/main.css';
// también podría ser un archivo sass y lo compila automaticamente, ejemplos:
// import './styles/main.scss';
// import './styles/vars.styl';

En el archivo de configuración de webpack webpack.config.js se anexa lo siguiente:

// webpack.config.js
const ...;
const MiniCssExtractPlugin = require('mini-css-extract-plugin');
module.exports = {
 ...,
 module: {
  rules: [
   {...},
    {
    test: /\.s?css|.styl|.sass$/i,
    use: [MiniCssExtractPlugin.loader,
    'style-loader',
    'css-loader',
    'stylus-loader',
    'sass-loader']
   }
  ]
 },
 plugins: {
  new ...({...}) ,
  new MiniCssExtractPlugin({
   filename: 'assets/[name].[contenthash].css'
  }),
 }
}

notas:

Es importante que se coloquen las reglas de CSS en ese orden: style-loader, css-loader, stylus-loader y sass-loader.

Existen conflictos entre style-loader y mini-css-extract-plugin por lo que se deberá comentar o eliminar style-loader de las reglas de CSS, también se puede desinstalar con el comando npm uninstall style-loader -D.

Por último se compila webpack y se obtienen todos los archivos de estilos combinados.

Otros loaders y optimizaciones.

Copy webpack

Si se necesita mover un archivo o directorio al proyecto final, se puede utilizar el plugin “copy-webpack-plugin”.

Se instala desde la terminal:

npm i copy-webpack-plugin -D

Se agrega la configuración en el archivo webpack.config.js:

// webpack.config.js
const ...;
const CopyPlugin = require('copy-webpack-plugin');
module.exports = {
 ...,
 module: {
  ...
 },
 plugins: {
  new ...({...}),
  new MiniCssExtractPlugin(),
   new CopyPlugin({
   patterns: [
    {
     // Se utiliza el path definido en las constantes del archivo y se especifica desde donde se importará el archivo o la carpeta. Con __dirname se especifica donde se esta posicionado actualmente, se coloca después la carpeta donde se aloja el proyecto y como tercer argumento se indica donde se encuentra el material que se copiará (puede ser una carpeta específica "assets/images" o una carpeta general "assets").
      // **Es mejor utilizar comillas dobles.**
     from: path.resolve(__dirname, "src", "assets"),
     // Se especifica hacia donde se copiarán los archivos o carpetas, en este caso se coloca una ruta que se creará dentro de la carpeta dist, todo archivo o carpeta que se encuentre en "assets" se colocará en la ruta que se coloque:
     to: "assets"
    }
   ]
}),
 }
}

Optimización de imágenes.

Se pueden importar las imágenes del proyecto generando una variable. No es necesario instalar ninguna dependencia, webpack ya lo tiene incluido y solo se debe agregar la siguiente configuración en el archivo de configuración de webpack webpack.config.js:

// webpack.config.js
const ...;
module.exports = {
 output: {
  ...,
   assetModuleFilename: 'assets/images/[hash][ext][query]'
 },
 ...,
 module: {
  rules: [
   {...},{...},
    {
    test: /\.(png|svg|jpg|jpeg|gif)$/i,
    type: "asset/resource"
   }
  ]
 },
 plugins: {...}
}

Después se importan todas las imágenes con sus direcciones al inicio del archivo .js o .jsx en donde se utilicen las imágenes, generalmente se utiliza en el archivo template.js, este se utiliza en react para crear el html.

// template.js
import nombre_de_variable from '../assets/images/nombre_de_imagen.png';
// Después se modifican las URL de las imágenes por las variables de la importación.
<img src="${nombre_de_variable}"/>

Se compila webpack de nuevo y listo.

Optimización de fuentes.

Cuando se utilizan fuentes externas, el proyecto se hace un poco más pesado y una buena práctica es descargar las fuentes en el proyecto para optimizar recursos. Webpack ayuda a optimizar aún más estos recursos.

Primero se deben descargar las fuentes que se ocupan en el proyecto en formato .woff y colocarlas en la carpeta assets creando una carpeta llamada fonts, una página recomendada para descargarlas es google-webfonts-helper.

Después se coloca el siguiente código en el archivo de estilos main.css:

// main.css
// En este caso se utilizará de ejemplo la fuente Ubuntu y las extensiones woff.
@font-face {
 font-family: "Ubuntu";
 font-style: normal;
 font-weight: 400;
 src: local(''),
  url("../assets/fonts/ubuntu-regular.woff2") format('woff2'),
  url("../assets/fonts/ubuntu-regular.woff") format('woff');
}

.body{
 font-family: 'Ubuntu';
}

Después se instalam las dependencias que ayudarán a optimizar la fuente con el comando:

npm install url-loader file-loader -D

Y a conticuación se coloca el siguiente código en el archivo de configuración webpack.config.js:

// webpack.config.js
const ...;
module.exports = {
 ...,
 module: {
  rules: [
   {...},{...},
    {
    test: /\.(woff|woff2)$/i,
    type: 'asset/resource',
    generator: {
      filename: 'assets/fonts/[hash][ext][query]',
    },
   }
  ]
 },
 plugins: {...}
}

Por último se compila el proyecto en webpack.

Optimizaciones.

Cuando el navegador entra a un sitio por primera vez, detecta datos del usuario y los guarda en la memoria cache como recursos que puede utilizar en cualquier momento. Por ello la siguiente vez que el navegador entre al mismo sitio será mucho más rápido porque estarán en la memoria cache listos para nadamás ser consultados.

La desventaja es que cuando sale una nueva versión del sitio web, los recursos tendrán un mismo nombre evitando que se descargue los nuevos cambios, por lo tanto, el usuario no recibirá los nuevos cambios. Para que no haya conflictos con la memoria cache, una vez que el proyecto esté en producción, es importante darles un hash a los recursos para cada nueva versión.

Para optimizar webpack se instalan las siguientes dependencias desde la terminal para minificar el código HTML y CSS:

npm i css-minimizer-webpack-plugin terser-webpack-plugin -D
# css-minimizer-webpack-plugin ⇒ comprime los archivos finales CSS.
# terser-webpack-plugin ⇒ minifica de una mejor forma.

Después, en el archivo de configuración webpack.config.js, se anexa el siguiente código:

// webpack.config.js
const ...;
const CssMinimizerPlugin = require('css-minimizer-webpack-plugin');
const TerserPlugin = require('terser-webpack-plugin');
module.exports = {
 ...
 Plugins: [ ... ],
  optimization: {
  minimize: true,
  minimizer: [
   new CssMinimizerPlugin(),
   new TerserPlugin()
  ]
 }
}

Por último se compila webpack en la terminal.

Alias.

Los alias permiten otorgar nombres a paths específicos evitando los paths largos, para crear un alias se agrega la siguiente configuración a webpack.config.js:

// webpack.config.js
module.exports = {  ...
 resolve: {
  extensions: ['.js'] ,
  alias: {
   '@nombre del alias': path.resolve(__dirname, 'ruta/de/carpetas/'),
   // la ruta que se desea guardar debe acabar con un slash, por ejemplo: src/images/
  }
 }, ...
}

Después se pueden modificar las rutas de los documentos de tipo .js o .jsx, generalmente se utiliza en el archivo template.js:

// template.js
import profilPicture from 'src/images/profilePicture.png'
import profilPicture from '@nombre del alias/profilePicture.png'

Por último se compila webpack y se verifica que se respeten los alias.

Variables de entorno.

Las variables de entorno son variables externas a un proyecto o aplicación que residen en el sistema operativo o en el contenedor de la aplicación que se está ejecutando. Se declaran usando mayúsculas y con formato SNAKE_CASE.

Estas variables son un espacio seguro donde se guardan datos sensibles del proyecto o aplicación. Como son sensibles, no se deben subir al repositorio y si se desea trabajar en el proyecto se deben pedir al líder del proyecto.

Para que webpack las reconozca y optimice se instala la siguiente dependencia:

npm install -D dotenv-webpack

Después, en la raíz del proyecto junto al README o al .gitignore, se crea un archivo llamado .env que es donde se almacenan las variables de entorno (no se sube al repositorio). De la misma manera se crea un archivo llamado .env.example que es en donde se colocan las mismas variables que en el archivo .env, pero sin asignarles un valor (se puede subir al repositorio).

// .env
API = https://.../
// .env.example
API =

En el archivo de configuración webpack.config.js se coloca la siguiente constante en las primeras lineas del código:

// webpack.config.js
const Dotenv = require('dotenv-webpack');

Y en la parte de los plugins del mismo archivo, se coloca lo siguiente:

// webpack.config.js
 plugins: [
  ... ,
   new Dotenv()
 ],

Después se modifican los archivos en donde se utilizan los valores de las variables de entorno de la siguiente manera:

const nombre = process.env.NOMBRE_VARIABLE;

Por último se compila el proyecto con webpack.

Deploy.

Es recomendable tener separadas las configuraciones del modo desarrollo y del modo producción, para ello se crean un archivo extra de configuración.

Modo desarrollo.

En la raíz del proyecto se crea un archivo llamado webpack.config.dev.js y se copia en éste todo el código del archivo webpack.config.js con exepción del fragmento optimization y al no tener esta sección de optimización, entonces también se eliminan las constantes declaradas al inicio del archivo que no se utilizan.

Se puede especificar al archivo que tiene la configuración especifica para el modo desarrollo con el siguiente fragmento de .código.

// webpack.config.dev.js
module.exports = { ...,
 output = {...},
  mode: 'development',
... }

Al compilar el proyecto desde la terminal se especifica que se utilice el archivo de desarrollo con el comando:

npx webpack --config webpack.config.dev.js

Modo producción.

El archivo de configuración webpack.config.js sirve para utilizar el modo producción, pero al compilarlo se debe asegurar de que el proyecto sea limpiado de archivos repetidos los cuales se fueron acumulando por compilaciones anteriores, para esto se instala clean-webpack desde la terminal:

npm install -D clean-webpack-plugin

En el archivo de configuración webpack.config.js se agrega lo siguiente:

// webpack.config.js
const { CleanWebpackPlugin } = require('clean-webpack-plugin');
 plugins: [
  ... ,
  new Dotenv(),
   new CleanWebpackPlugin(),
 ],

Al compilar el proyecto desde la terminal se especifica que se utilice el archivo de producción con el comando:

npx webpack --mode production --config webpack.config.js

Hay una alternativa en implementada recientemente en la cual se coloca clean: true y limpia el directorio cada que se compile.

// webpack.config.js
module.exports = {
 ...
 output: {
   clean: true,
 },
};

Webpack watch.

El modo watch hace que el proyecto se compile de forma automática al realizar cambios en él. La desventaja esta en que si se aplica el webpack-watch en modo producción, se tomara más tiempo entre más grande sea el proyecto porque se optimizan los recursos, en cambio, en el modo desarrollo se salta ese paso y es más rápida la compilación.

Para activarlo se coloca el siguiente código en el archivo de configuración de desarrollo webpack.config.dev.js

// webpack.config.dev.js
module.exports = {
 ...
 output: {...},
  watch: true,
};

Después se compila el modo desarrollo desde la terminal y la terminal se quedará escuchando los cambios del proyecto para compilarlo con cada cambio que se haga. Para detener el webpack-watch se presiona control + C.

Otra manera es compilar desde la terminal e indicar que se active el webpack-watch mediante el comando.

npx webpack --watch --config webpack.config.dev.js

Deploy a Netlify.

Para que el proyecto se visualice en internet se puede subir a Netlify.

Primero se debe crear en la raíz del proyecto el archivo netlify.toml y se especifica en éste atraves de un buld, que carpeta del proyecto se debe publicar y el comando que se utilizará para desplegar la aplicación.

// netlify.toml
[build]
 publish: "dist"
 command: "npm run build"

A través de la página de Netlify se elige el repositorio, la rama y también el comando y la carpeta que se especifica en el archivo netlify.toml.

Después se crea, en la raíz del proyecto, la carpeta scripts y adentro de esta carpeta se crea el archivo create-env.js y en este script se coloca el siguiente código:

// create-env.js
// Se llama un recurso para crear archivos de forma local.
const fs = require('fs');
// Se crea el archivo .env y se le agrega de contenido el ejemplo de variable de entorno antes visto. fs.writeFileSync('./.env', `API=${process.env.API}\n`);

Para que Netlify acepte las variables de entorno fue necesario crear el archivo anterior además de que en la página de Netlify se deben agregar en las sección Build & deploy en la parte de variables de entorno se agregan las variables del proyecto y así solamente Netlify podrá ver y usar estas variables.

Por último se compila el proyecto colocando primero el script creado:

npx node ./scripts/create-env.js && webpack --mode production --config webpack.config.js

Herramientas de desarrollo.

Webpack Dev Server.

Esta herramienta permite ver los cambios en tiempo real en un navegador y es capaz de hostear aplicaciones con frameworks y librerías de JavaScript como Vue o React.

Se debe desactivar el watch (en el archivo de configuración de desarrollo -> watch:false) o cualquier otra herramienta que esté escuchando el código cuando se compila la aplicación.

Para utilizarla se instala con el comando:

npm install -D webpack-dev-server

Esta herramienta ayudara a ver cambios al momento de desarrollar la aplicación, por lo que se deben agregar las configuraciones de esta herramienta en el archivo de desarrollo webpack.config.dev.js:

// webpack.config.dev.js
module.exports = {
 ... ,
  devServer: {
  static: {
   directory: path.join(__dirname, "dist"),
   watch: true,
  },
  //Para observa los cambios en todos los archivos y actualizar el navegador:
  watchFiles: path.join(__dirname, "./**"),
  // Para comprimir:
  compress: true,
  // Para obtener un historial del navegador:
  historyApiFallback: true,
  port: 3000,
  // Para abrir en el navegador:
  open: true,
 }
}

Como en los comandos anteriores, este se puede agregar en el archivo package.json, en la sección de scripts para evitar teclear comandos muy largos:

"scripts": {
 ... ,
  "start": "webpack serve --config webpack.config.dev.js"
}

Y en la terminal se colocaría el siguiente comando:

npm run start
// en vez de teclear:
npx webpack serve --config webpack.config.dev.js

Entre tanto, en la terminal devolverá una URL para visualizar la aplicación con sus cambios.

Webpack Bundle Analyzer.

En un proyecto se debe revisar su impacto en tamaño, para ayudar a ésto, webpack ofrece un paquete para poder verificar y analizar el tamaño del bundle final. Para poder utilizar esta herramienta se tiene que instalar desde la terminal con el comando:

npm install -D webpack-bundle-analyzer

Después en el archivo de configuración webpack.config.dev.js se anexa lo siguiente:

const ... ,
const {BundleAnalyzerPlugin} = require('webpack-bundle-analyzer').BundleAnalyzerPlugin;
module.exports = {
 plugins: [
  new ... ,
   new BundleAnalyzerPlugin(),
 ],
 ...
}

Con el siguiente comando webpack analizará el proyecto y guardará un reporte, de las dependencias y recursos que se utilizan, en un archivo de tipo .json llamado stats:

npx webpack --profile --json > stats.json

Por último, con el siguiente comando webpack lee el reporte y lo presenta en una página web:

npx webpack-bundle-analyzer stats.json

Webpack Dev Tools.

Sirve para hacer debuggin al código de una forma muy fácil.

Primero se anexa el siguiente código al archivo webpack.config.dev.js:

module.exports = {
 entry: ...,
 output: {...},
 mode: ... ,
  devtool: 'source-map',
 resolve: {...},
 ...
}

Después se compila el proyecto en el modo desarrollo desde la terminal:

npx webpack --mode development

Se abre el proyecto con la herramienta de Visual studio Live server y en el navegador en el inspector de elementos, en la sección source se puede ver el código y si se da clic sobre el numero de línea de alguna línea de código se podrá visualizar la pieza de código separada que involucra a esa línea cliqueada.
